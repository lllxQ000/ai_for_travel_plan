# 召回率与准确率在 RAG 评估中的含义及使用方法

在 RAG（检索增强生成）系统中，评估通常分为两个阶段：**检索阶段**（Retrieval）和**生成阶段**（Generation）。召回率（Recall）和准确率（Precision）是检索阶段最核心的指标，用来衡量检索模块是否找到了真正相关的文档块。而生成阶段则需要另外的指标（如忠实度、答案正确性等）。

---

## 一、什么是召回率与准确率（检索阶段）

假设我们有一个测试集，包含 N 个问题，每个问题对应一组人工标注的相关文档块（ground truth）。

- **召回率（Recall）**：  
  ![image-20260324172422042](https://pub-64ffffada5d441c18d41525eba7abb52.r2.dev/PicGo/image-20260324172422042.png)
  它衡量检索系统“找全”的能力。如果召回率低，说明很多相关块没有被检索到，可能导致答案遗漏关键信息。
  
- **准确率（Precision）**：  
  ![image-20260324172519839](https://pub-64ffffada5d441c18d41525eba7abb52.r2.dev/PicGo/image-20260324172519839.png)
  它衡量检索系统“找对”的能力。如果准确率低，说明检索结果中包含大量不相关的噪声，可能误导生成模型。

在 RAG 中，我们通常关心 **Precision@k** 和 **Recall@k**（针对前 k 个检索结果），因为检索器会返回固定数量的块（如 top‑5），生成模型只基于这些块回答。

---

## 二、RAG 评估通常依赖哪些指标？

**不是只依赖召回率和准确率**，而是需要一个指标体系。常见的 RAG 评估维度包括：

### 1. 检索模块评估
- **Recall@k / Precision@k**：直接衡量检索质量。
- **MRR（平均倒数排名）**：考虑相关块的排序位置。
- **NDCG（归一化折损累积增益）**：对排序质量更精细的评估。

### 2. 生成模块评估
- **忠实度（Faithfulness）**：生成的答案是否严格基于检索到的上下文，有没有“幻觉”。
- **答案相关性（Answer Relevance）**：答案是否切题，解决了用户问题。
- **上下文相关性（Context Relevance）**：检索到的上下文本身与问题的相关程度。
- **BLEU / ROUGE**：与参考答案的相似度（常用于有标准答案的场景）。
- **人工评价**：对答案的实用性、流畅性进行打分。

### 3. 端到端评估
- **正确率（Correctness）**：答案是否正确（可用 LLM 作为裁判或人工标注）。
- **用户满意度**：通过点赞/点踩收集。

---

## 三、在项目中如何实际做 RAG 评估？

由于你是学生项目，时间和资源有限，建议采用以下轻量级评估方法：

### 1. 构建一个小型测试集
- 准备 20~50 个有代表性的问题。
- 人工标注每个问题对应的正确文档块（从你的知识库中找出）。
- 记录期望的答案（可选）。

### 2. 计算检索指标
运行检索系统，对每个问题取 top‑k 结果（如 k=5），计算：
- **Recall@5**：前 5 个结果中包含的相关块数 / 总相关块数。
- **Precision@5**：前 5 个结果中相关块数 / 5。

### 3. 评估生成答案
- **人工评估**：邀请同学或老师对每个问题的答案打分（1~5 分），综合评判。
- **辅助 LLM 评估**：使用 GPT-4 或 Qwen 等模型对答案进行评分（提示词设计要规范）。

### 4. 对比实验
如果有条件，设置一个对照组（例如直接使用 Qwen3.5 而不检索），对比两种方式在测试集上的表现，证明 RAG 的有效性。

## 一、`rag_pipeline.py` —— RAG 基础工具

该模块提供了检索、重排序、自动合并、查询扩展等原子能力，供上层工作流调用。

### 1. 主要函数功能
- **`_get_rerank_endpoint()`**：构建外部 Rerank API 的 URL。
- **`_merge_to_parent_level(docs, threshold)`**：将叶子块按父块 ID 分组，若某父块下的子块数量 ≥ 阈值，则用父块文本替换这些子块。递归调用实现 L3→L2 和 L2→L1 的两级合并。
- **`_auto_merge_documents(docs, top_k)`**：调用两次 `_merge_to_parent_level`（先 L3→L2，再 L2→L1），返回合并后的文档列表及合并统计信息。
- **`_rerank_documents(query, docs, top_k)`**：将检索结果发送给外部 Rerank API（如 Jina），获得精排后的文档列表。包含异常降级逻辑。
- **`step_back_expand(query)`**：生成退步问题及其答案，返回扩展后的查询字符串。
- **`generate_hypothetical_document(query)`**：生成 HyDE 假设文档，用于二次检索。
- **`retrieve_documents(query, top_k)`**：**主入口**  
  1. 尝试混合检索（稠密+稀疏），若失败则降级为纯稠密检索。  
  2. 调用 `_rerank_documents` 精排。  
  3. 调用 `_auto_merge_documents` 进行自动合并。  
  4. 返回最终文档列表和丰富的元数据（用于前端展示和追踪）。

### 2. 设计亮点
- **健壮性**：混合检索失败时自动降级为稠密检索，避免整体失败。
- **可配置性**：通过环境变量控制合并阈值、是否启用自动合并、检索层级等。
- **可观测性**：返回的元数据包含检索模式、重排信息、合并统计等，便于后续分析和前端展示。

## 二、`rag_utils.py` —— 基于 LangGraph 的 RAG 工作流

该模块使用 LangGraph 构建了一个有状态、可分支的 RAG 流程，将检索、评分、重写、二次检索组织成有向图。

### 1. 状态定义
`RAGState` 是一个 TypedDict，记录了问题、查询、文档列表、上下文、路由决策、扩展查询、trace 等。

### 2. 节点函数
- **`retrieve_initial(state)`**  
  调用 `retrieve_documents` 进行初次检索，并发射步骤事件（通过 `emit_rag_step`）。返回文档、格式化的上下文以及初始 trace。
- **`grade_documents_node(state)`**  
  使用评分模型（LLM 结构化输出）判断当前检索结果是否相关。若相关则路由到 `generate_answer`（END），否则路由到 `rewrite_question`。
- **`rewrite_question_node(state)`**  
  首先用路由器模型选择扩展策略（`step_back` / `hyde` / `complex`），然后调用 `step_back_expand` 或 `generate_hypothetical_document` 生成扩展查询。返回扩展后的查询及相关中间结果。
- **`retrieve_expanded(state)`**  
  根据策略，使用扩展查询（或 HyDE 假设文档）再次调用 `retrieve_documents`，并将多路结果合并去重，更新 trace。

### 3. 图构建
```
retrieve_initial → grade_documents
                ├─ (grade=yes) → END
                └─ (grade=no) → rewrite_question → retrieve_expanded → END
```

### 4. 设计亮点
- **模块化**：每个节点职责单一，便于单独测试和调试。
- **条件路由**：评分门控实现动态决策，避免无效检索浪费。
- **实时反馈**：每个节点都通过 `emit_rag_step` 推送进度，前端可实时显示思考步骤。
- **可扩展性**：新增节点或修改策略只需调整图结构，不影响其他部分。

## 三、为什么选择 LangChain 而不是其他 Agent 框架？

项目采用 LangChain（含 LangGraph）作为核心框架，主要原因如下：

### 1. **生态完整，RAG 支持成熟**
LangChain 提供了文档加载、分块、向量存储（与 Chroma、FAISS 等无缝集成）、检索器、Agent、记忆管理等全套组件。本项目需要将多个环节串联，LangChain 的抽象让开发变得高效。例如：
- `VectorStore` 统一了不同向量库的接口。
- `create_agent` 内置了 ReAct 循环，免去手写工具调用逻辑。

### 2. **LangGraph 天然适合复杂流程**
RAG 流程包含分支（评分门控）和多步（重写后二次检索），LangGraph 以图的形式描述状态机，代码清晰且易于扩展。相比其他框架（如 LlamaIndex 的 `QueryPipeline`），LangGraph 对条件分支和状态管理的支持更灵活，且与 LangChain 生态无缝衔接。

### 3. **工具调用与 Agent 能力**
项目需要 Agent 根据问题决定调用知识库检索或天气查询。LangChain 的 Agent 抽象（`create_agent`）允许快速绑定自定义工具，并提供流式输出支持。其他框架（如 AutoGen）更侧重多智能体协作，而本项目只需单个 Agent，LangChain 足够且轻量。

### 4. **社区与文档**
LangChain 是目前最流行的 LLM 应用框架，遇到问题容易找到解决方案。对于学生项目，丰富的教程和示例能显著降低学习成本。

### 5. **与现有技术栈契合**
后端使用 Flask，LangChain 的同步/异步接口均可配合。虽然 LangChain 在异步框架（如 FastAPI）中优势更明显，但通过后台线程+队列，本项目仍实现了流式输出和实时步骤推送。LangChain 提供的 `agent.astream` 统一了流式生成接口，无需关心底层 LLM 的流式细节。

### 6. **避免重复造轮子**
从零实现工具调用、记忆管理、流式输出、查询重写等会耗费大量时间。LangChain 提供了开箱即用的组件（如 `ConversationSummaryBufferMemory`、`create_agent`），使团队能集中精力优化 RAG 核心算法（如混合检索、自动合并、评分门控），这些正是项目的创新点。

## 四、总结

- **`rag_utils.py`** 封装了检索、精排、自动合并、查询扩展等底层能力，是 RAG 系统的“工具箱”。
- **`rag_graph.py`** 利用 LangGraph 构建了可观测、有状态的工作流，实现了评分门控和智能重写，是 RAG 系统的“大脑”。
- **选择 LangChain** 是因为其生态完整、LangGraph 适合复杂流程、工具调用灵活、社区支持好，能够快速搭建高质量 RAG 系统，符合学生项目对开发效率和稳定性的需求。
