# 从零开始构建 SuperMew RAG 项目的软件开发全流程

本指南将按照**需求分析 → 软件架构 → 软件测试**三大阶段，结合本项目实际，详细阐述每个阶段的目标、方法以及具体的“代码解决路径”（即从零到一的实现步骤）。阅读后你将清晰了解如何系统化地开发一个基于 LangChain + Flask + Chroma 的 RAG 智能问答系统。

---

## 一、需求分析阶段

### 1.1 目标
明确“我们要做什么”，将模糊的用户期望转化为具体、可衡量的功能需求和约束。

### 1.2 方法
- **业务调研**：了解 OTA 平台 AI 助手的现状痛点（信息滞后、回答不精准）。
- **用户画像**：典型用户为旅游规划者、客服人员，需要快速获取最新产品信息、活动规则。
- **功能拆解**：
  - 知识库管理：文档上传、分块、向量化存储
  - 智能问答：基于检索增强的对话
  - 实时反馈：检索步骤可视化、流式输出
  - 多轮记忆：会话上下文保持
- **非功能需求**：响应时间（首字 ≤3s）、准确率（≥85%）、可扩展性、成本可控。

### 1.3 本项目中的代码解决路径

1. **用文字或原型工具绘制用户故事与流程图**  
   - 产出：`docs/user_stories.md`、`docs/flowchart.png`
   - 明确用户与系统的交互流程：用户提问 → 系统检索 → 生成答案 → 流式返回

2. **定义数据模型（Schema）**  
   - 在 `schemas.py` 中定义 Pydantic 模型，作为前后端交互的契约。
   ```python
   from pydantic import BaseModel
   class ChatRequest(BaseModel):
       message: str
       user_id: str = "default_user"
       session_id: str = "default_session"
   ```

3. **编写 API 接口清单**  
   - 用注释或文档记录每个接口的用途、请求/响应格式。例如：
   ```
   POST /chat/stream     # 流式聊天
   POST /documents/upload # 上传文档
   GET  /sessions/{user_id} # 列出会话
   ```

4. **搭建最小原型验证可行性**  
   - 快速实现一个最简单的 Flask 应用，调用一个 LLM 返回固定回答，确保前后端通信正常。  
   - 这一阶段**不涉及** RAG 和 Chroma，仅验证技术链路。

## 二、软件架构阶段

### 2.1 目标
确定系统的整体结构、模块划分、技术选型、数据流，确保后续开发有序进行。

### 2.2 方法
- **分层架构**：表示层（前端）、业务逻辑层（Agent + RAG）、数据访问层（Chroma、存储）
- **模块划分**：按功能拆分为独立模块（如 embedding、rag_utils、agent、tools 等）
- **技术选型**：Flask + LangChain + Chroma + 阿里云模型
- **数据流设计**：用户消息 → Agent → 工具 → RAG Pipeline → 检索 → 生成 → 流式返回

### 2.3 本项目中的代码解决路径

1. **设计项目目录结构**（如前文所示），每个模块职责单一。

2. **实现基础通信层**  
   - `app.py`：Flask 应用入口，注册路由。
   - 实现 `/chat/stream` 的基础框架，先返回静态测试数据，确保 SSE 可用。

3. **逐步实现核心模块**（按依赖顺序）  
   - **Step 1: 向量化服务** (`embedding.py`)  
     - 封装阿里云嵌入 API，提供 `get_embeddings(texts)` 和 `get_sparse_embedding(text)` 方法。
     - 编写单元测试：验证返回的向量维度和类型正确。

   - **Step 2: 文档处理** (`document_loader.py`)  
     - 实现文档加载（PDF/Word）和三级分块逻辑。
     - 返回叶子块列表和父块列表。

   - **Step 3: 向量存储** (`chroma_client.py`)  
     - 初始化 Chroma 客户端，创建 `doc_chunks` 和 `parent_chunks` 集合。
     - 实现添加块、查询块的方法。

   - **Step 4: 检索与合并工具** (`rag_utils.py`)  
     - 实现 `retrieve_documents`：向量检索 → Rerank → Auto‑merging。
     - 实现 `step_back_expand`、`generate_hypothetical_document` 等辅助函数。

   - **Step 5: RAG 工作流** (`rag_pipeline.py`)  
     - 使用 LangGraph 构建状态图，串联初次检索、评分、重写、二次检索。
     - 在每个节点中调用 `emit_rag_step` 发送实时步骤。

   - **Step 6: 工具封装** (`tools.py`)  
     - 将 `rag_pipeline` 包装为 `search_knowledge_base` 工具。
     - 实现 `emit_rag_step` 的跨线程调度（使用 `call_soon_threadsafe`）。

   - **Step 7: Agent 与对话** (`agent.py`)  
     - 使用 LangChain 的 `create_agent` 构建 Agent，绑定工具。
     - 实现流式对话生成，将 token 和步骤事件推送到队列。

   - **Step 8: 会话存储** (`conversation_storage.py`)  
     - 基于 Chroma 实现 `save_message`、`load_messages`、`list_sessions` 等方法。

   - **Step 9: 前端集成**  
     - 编写 `index.html` 和 `script.js`，实现 SSE 接收、打字机效果、中止功能。

4. **整合所有模块到 `app.py`**  
   - 在路由中调用 `agent.chat_with_agent_stream` 并返回流式响应。

## 三、软件测试阶段

### 3.1 目标
确保系统功能正确、性能达标、容错健壮。

### 3.2 方法
- **单元测试**：测试独立函数（如分块、向量化、合并逻辑）。
- **集成测试**：测试模块间交互（如检索+重排、Agent+工具）。
- **端到端测试**：模拟用户操作，验证完整流程。
- **RAG 专项评估**：计算召回率、准确率、忠实度等指标。

```python
import requests
import json

def test_chat_stream():
    resp = requests.post('http://localhost:5000/chat/stream', json={
        "message": "北海有哪些景点？"
    }, stream=True)
    full_text = ""
    for line in resp.iter_lines():
        if line:
            if line.startswith(b'data: '):
                event = json.loads(line[6:])
                if event['type'] == 'content':
                    full_text += event['content']
                elif event['type'] == 'rag_step':
                    print("Step:", event['step']['label'])
    assert len(full_text) > 0
```

#### 3.3.4 RAG 专项评估

**构建测试集**  
- 准备 30-50 个问题，人工标注每个问题对应的正确文档块 ID。  
- 存储为 `test_questions.json`。

**计算召回率与准确率**  
```python
def evaluate_retrieval():
    with open('test_questions.json') as f:
        test_data = json.load(f)
    total_recall = 0
    total_precision = 0
    for item in test_data:
        question = item['question']
        ground_truth = set(item['chunk_ids'])
        result = retrieve_documents(question, top_k=5)
        retrieved_ids = [doc['chunk_id'] for doc in result['docs']]
        relevant_retrieved = set(retrieved_ids) & ground_truth
        recall = len(relevant_retrieved) / len(ground_truth) if ground_truth else 0
        precision = len(relevant_retrieved) / len(retrieved_ids) if retrieved_ids else 0
        total_recall += recall
        total_precision += precision
    avg_recall = total_recall / len(test_data)
    avg_precision = total_precision / len(test_data)
    print(f"Recall@5: {avg_recall:.3f}, Precision@5: {avg_precision:.3f}")
```

## 四、总结与建议

1. **渐进式开发**：不要试图一次性实现所有功能，而是按照“基础通信 → 向量化 → 检索 → 生成 → 流式 → 优化”的节奏推进。
2. **保持可测试性**：每个模块尽量编写单元测试，集成测试尽早介入。
3. **文档同步**：使用 Markdown 记录架构决策、API 设计、测试用例，便于团队协作和答辩准备。
4. **版本控制**：使用 Git 管理代码，每完成一个阶段打一个 tag，便于回溯。

按照上述路径，你将能系统化地完成一个高质量的 RAG 项目，并在复试中自信地阐述你的设计思路、实现细节与测试验证过程。祝你项目顺利！
