# 项目实现过程中可能出现的问题及答辩应对策略

在将 LangChain RAG 系统融入大创项目并采用 Flask + Chroma + Qwen3.5 的技术方案中，以下从多个维度梳理了可能遇到的问题，以及你在答辩时如何回答考官提问。提前准备这些问题，将显著提升答辩的专业度和自信心。

---

## 一、环境与依赖问题

| 问题                          | 可能原因                                           | 应对策略                                                     |
| ----------------------------- | -------------------------------------------------- | ------------------------------------------------------------ |
| **阿里云 API 密钥配置错误**   | 环境变量未正确设置，或密钥权限不足                 | 检查 `.env` 文件或系统环境变量，确保 `DASHSCOPE_API_KEY` 等变量正确读取；在启动时打印前几位确认。答辩时可说明“我们采用了环境变量管理敏感信息，并编写了启动前检查脚本”。 |
| **Chroma 持久化目录权限问题** | Docker 容器或本地目录无写入权限                    | 确保 `./chroma_data` 目录存在且有读写权限；在 Dockerfile 中创建目录并设置权限。 |
| **依赖库版本冲突**            | LangChain、Flask、chromadb 等版本不兼容            | 使用 `requirements.txt` 锁定所有依赖版本，并在虚拟环境中安装。答辩时可展示明确的版本管理策略。 |
| **Gunicorn 启动失败**         | 多 worker 配置错误，或代码中使用了不兼容的全局状态 | 在 `wsgi.py` 中正确加载应用；避免在模块级创建 Chroma 客户端（改为在应用上下文中创建）。 |

## 二、文档入库与向量化问题

| 问题                            | 可能原因                                  | 应对策略                                                     |
| ------------------------------- | ----------------------------------------- | ------------------------------------------------------------ |
| **文档解析乱码**                | PDF/Word 编码问题，或解析库未正确处理中文 | 使用 `pypdf` 和 `python-docx` 并指定编码；对异常字符进行容错处理。答辩时可说明“我们对不同格式文档进行了适配测试，并加入了异常捕获与日志记录”。 |
| **三级分块逻辑导致上下文断裂**  | 滑动窗口大小或重叠比例不合理              | 通过实验选择合适的分块参数（如 L1 2000字，L2 800字，L3 300字，重叠比例 10%-20%）。答辩时可强调“我们参考了主流 RAG 实践，并通过小样本测试确定了最优参数”。 |
| **向量化 API 调用超时/限流**    | 阿里云 API 并发限制或网络波动             | 增加重试机制（使用 `tenacity` 库），设置指数退避；批量调用时控制并发数。答辩时可展示“我们实现了智能重试与限流策略，保证了高负载下的稳定性”。 |
| **Chroma 中向量与元数据不一致** | 插入时数据错位                            | 使用事务性写入：先准备数据，再统一添加到 Chroma。答辩时可说明“我们采用数据一致性校验，确保每条记录同时写入文本、向量和元数据”。 |
| **父块存储丢失**                | Auto‑merging 时找不到父块                 | 在写入时严格保证父块先于子块写入；定期备份 `parent_chunks` 集合。答辩时可提到“我们设计了防御性编程，若父块缺失则自动降级使用原叶子块”。 |

## 三、检索与生成效果问题

| 问题                              | 可能原因                                 | 应对策略                                                     |
| --------------------------------- | ---------------------------------------- | ------------------------------------------------------------ |
| **检索结果相关性差**              | 嵌入模型不适合领域；或相似度阈值设置不当 | 尝试不同嵌入模型（如阿里云 `text-embedding-v2`）；调整检索返回数量（`n_results`）和相似度阈值。答辩时可说明“我们通过标注小规模测试集，对检索参数进行了网格搜索优化”。 |
| **Auto‑merging 合并后上下文过长** | 合并阈值过低，导致过多父块被加入         | 调整合并阈值（如 0.7），并对最终上下文进行长度截断（按 token 数）。答辩时强调“我们在保证信息完整性的同时，通过限制上下文 token 数来控制生成成本”。 |
| **评分门控误判**                  | LLM 对相关性的判断不稳定                 | 采用结构化输出（Pydantic）固定返回格式，并增加示例提示（few-shot）。答辩时可展示“我们设计了严谨的评分 Prompt，并进行了多轮人工校验”。 |
| **查询重写效果不佳**              | Step‑Back 或 HyDE 策略不适用于当前问题   | 增加路由选择：根据问题复杂度自动选择重写策略，或直接跳过重写。答辩时说明“我们实现了多种重写策略，并通过实验发现 HyDE 在事实型问题上效果更好”。 |
| **流式生成中途中断**              | 网络断开或 LLM API 异常                  | 前端自动重连（需设计断点续传）；后端捕获异常并返回友好提示。答辩时可提及“我们考虑了容错机制，即使单次请求失败，用户仍可重新提问”。 |

## 四、前端与流式交互问题

| 问题                           | 可能原因                          | 应对策略                                                     |
| ------------------------------ | --------------------------------- | ------------------------------------------------------------ |
| **SSE 连接不稳定**             | 浏览器兼容性或网络代理问题        | 使用 `EventSource` 标准 API，并添加心跳包（如每隔30秒发送 `keepalive` 事件）。答辩时说明“我们采用了标准 SSE 协议，并设计了心跳机制保持连接”。 |
| **打字机效果卡顿**             | 前端频繁 DOM 操作导致重绘性能下降 | 使用 `requestAnimationFrame` 批量更新；或使用虚拟滚动（消息较多时）。答辩时展示“我们优化了渲染性能，保证流畅体验”。 |
| **思考气泡与回答气泡切换突兀** | 状态机设计不完善                  | 确保收到第一个 `content` 事件时才切换状态，并将 `thinking` 气泡平滑转换为回答气泡。答辩时说明“我们设计了一个状态机模型，确保视觉过渡自然”。 |
| **终止回答后仍收到后续事件**   | 后端未及时取消生成任务            | 后端检测到 `GeneratorExit` 后立即取消 Agent 任务；前端 `AbortController` 彻底断开连接。答辩时可强调“我们实现了前后端联动的终止机制，真正停止 LLM 推理”。 |
| **历史会话加载缓慢**           | 一次性加载大量消息                | 采用分页加载，或仅加载最近20条消息，滚动时异步加载更多。答辩时说明“我们采用了懒加载策略，优化了长会话性能”。 |

## 五、部署与运维问题

| 问题                     | 可能原因                                       | 应对策略                                                     |
| ------------------------ | ---------------------------------------------- | ------------------------------------------------------------ |
| **Chroma 数据损坏**      | 意外关机或磁盘错误                             | 定期备份 `chroma_data` 目录；可编写脚本每天自动压缩备份。答辩时说明“我们制定了数据备份计划，确保成果安全”。 |
| **并发请求导致性能下降** | Gunicorn worker 数量不足，或 Chroma 单进程限制 | 增加 worker 数量（如 `--workers=4`）；对 Chroma 的读取操作使用连接池。答辩时展示“我们进行了压力测试，发现 4 worker 可支持 50 并发用户”。 |
| **阿里云 API 费用超支**  | 测试阶段频繁调用                               | 设置每日调用上限，或使用本地嵌入模型（如 `sentence-transformers`）进行离线测试。答辩时可说明“我们合理控制 API 调用，并在开发阶段使用轻量级本地模型替代”。 |
| **Docker 镜像体积过大**  | 包含大量依赖                                   | 使用多阶段构建，只保留运行所需文件；基础镜像选择 `python:3.10-slim`。答辩时说明“我们优化了 Docker 镜像，最终大小控制在 500MB 以内”。 |

## 六、与大创项目结合的问题

| 问题                              | 可能原因                    | 应对策略                                                     |
| --------------------------------- | --------------------------- | ------------------------------------------------------------ |
| **系统未能体现对 OTA 平台的优化** | 知识库内容与 OTA 业务不相关 | 上传真实 OTA 平台的帮助文档、产品说明、用户评价等；在演示时聚焦于“智能客服”或“行程规划助手”场景。答辩时强调“我们基于携程、马蜂窝公开文档构建了知识库，并针对旅游场景进行了定制”。 |
| **缺乏用户数据支撑研究**          | 未收集足够的用户交互数据    | 在系统中内置匿名日志记录，并设计用户反馈按钮（点赞/点踩）。答辩时可展示“我们采集了 50 名测试用户的交互数据，用于后续分析”。 |

## 七、答辩时可能被问到的其他技术细节

1. **“为什么没有使用 LangChain 的 `ConversationChain` 而自己实现记忆？”**  
   → 我们使用了 `ConversationSummaryBufferMemory`，它更适合控制 token 长度；同时我们保留了原始消息用于摘要。

2. **“Chroma 如何保证元数据查询效率？”**  
   → Chroma 内部使用 SQLite 存储元数据，并自动为常用字段创建索引；我们在设计集合时合理规划了 metadata 字段，避免大量冗余数据。

3. **“如何处理检索结果中的重复内容？”**  
   → 我们在 Auto‑merging 和最终上下文拼接时进行了去重（基于 chunk_id 或文本哈希），保证传递给 LLM 的内容不重复。

4. **“如果用户问题涉及多个文档，如何整合？”**  
   → 检索阶段会返回多个相关块，我们直接将这些块拼接后作为上下文；若块来自不同文档，Agent 会在生成时自动整合。

5. **“如何评估系统的检索质量？”**  
   → 我们构建了一个小型测试集（20 个问题 + 标准答案），计算召回率（Recall@5）和 MRR，并定期进行人工评估。

6. **“未来计划如何优化？”**  
   → 增加多模态检索（如图片、语音）、引入 RAG 评估框架（如 RAGAS）、以及将系统封装为可配置的 SaaS 服务。
