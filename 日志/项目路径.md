# 基于 LangChain RAG 的 OTA 平台 AI 助手优化项目整体路径（Chroma 统一存储版）

> 本方案将 LangChain RAG 能力融入“基于 AIGC 的 OTA 产品用户使用体验研究与产品优化设计”大创项目，采用 **Flask** 后端、**Chroma** 作为唯一数据库（同时存储向量与结构化数据）、**Qwen3.5** 作为生成模型、**阿里云向量模型** 作为嵌入模型，构建具备实时检索、流式输出、会话记忆的智能问答系统，解决 OTA 平台 AI 助手信息滞后问题。

---

## 一、需求分析

（同前，保留原需求分析，略作调整）

### 1.1 业务目标
- 解决 OTA 平台 AI 助手因知识库更新不及时导致回答过时的问题。
- 提升回答质量：通过检索增强生成，使回答更加精准、可溯源、个性化。
- 为后续用户体验研究提供可观测的交互数据。

### 1.2 功能需求
| 模块           | 需求描述                                                     |
| -------------- | ------------------------------------------------------------ |
| 知识库管理     | 管理员可上传/删除旅游产品说明、活动文档；文档自动分块、向量化并存入 Chroma。 |
| 智能问答       | 用户提问后，系统执行混合检索（语义检索 + 关键词检索，但 Chroma 本身主要支持语义，可通过元数据过滤模拟关键词，或结合外部 BM25）→ 精排 → 生成答案（流式输出）。 |
| 实时检索可视化 | 前端展示检索步骤，增强交互透明度。                           |
| 多轮对话记忆   | 支持会话上下文，自动摘要历史消息，控制 token 长度。          |
| 用户反馈收集   | 收集点赞/点踩，用于评估与优化。                              |

### 1.3 非功能需求
- 响应时间：首字 ≤ 3 秒（因模型在云端），完整回答 ≤ 10 秒。
- 准确率：关键事实准确率 ≥ 85%（初期）。
- 可扩展：支持数千份文档，检索延迟可控。

## 二、概念结构设计

### 2.1 核心实体关系
- **用户** (user_id, 昵称, 注册时间) —— 1 : n —— **会话**
- **会话** (session_id, 标题, 创建时间, 用户ID) —— 1 : n —— **消息**
- **消息** (msg_id, 角色, 内容, 时间戳, 会话ID) —— n : 1 —— **会话**
- **文档** (doc_id, 文件名, 上传时间, 状态) —— 1 : n —— **分块**
- **分块**(改用CSV数据后不再需要分块，每一条记录就是一个块) (chunk_id, 级别 L1/L2/L3, 文本, 父块ID, 文档ID) —— n : 1 —— **文档**
- **检索记录** (record_id, 查询文本, 命中块列表, 耗时, 消息ID) —— 关联消息

### 2.2 统一存储（Chroma）设计理念
所有实体（用户、会话、消息、文档、分块）均作为 **Chroma Collection** 中的文档存储。每个 Collection 对应一类实体，文档中包含：
- `id`：唯一标识符
- `embedding`：可选，用于支持向量检索的实体（如文档分块）才需要
- `metadata`：键值对，存储该实体的所有属性（如 user_id, session_id, role, timestamp 等）
- `document`：文本内容（对于分块是块文本，对于消息是消息内容）

这样无需额外的关系数据库，完全由 Chroma 承担数据存储与检索功能。

## 三、逻辑结构设计

### 3.1 Chroma Collection 设计

| Collection 名称  | 用途                                           | 是否包含向量 | 主要 metadata 字段                                           |
| ---------------- | ---------------------------------------------- | ------------ | ------------------------------------------------------------ |
| `users`          | 存储用户信息                                   | 否           | `user_id`, `nickname`, `created_at`                          |
| `sessions`       | 存储会话信息                                   | 否           | `session_id`, `user_id`, `title`, `created_at`               |
| `messages`       | 存储对话消息                                   | 否           | `msg_id`, `session_id`, `role`, `timestamp`, `feedback`（可选） |
| `documents`      | 存储文档元数据                                 | 否           | `doc_id`, `filename`, `upload_time`, `status`                |
| `chunks`         | 存储文档分块（仅 L3 叶子块）                   | **是**       | `chunk_id`, `doc_id`, `level`, `parent_id`, `text_preview`（用于显示） |
| `parent_chunks`  | 存储 L1/L2 父块（不含向量，用于 Auto‑merging） | 否           | `chunk_id`, `doc_id`, `level`, `parent_id`, `text`           |
| `retrieval_logs` | 存储检索记录                                   | 否           | `log_id`, `msg_id`, `query`, `retrieved_chunks`（JSON 字符串）, `latency_ms` |

**说明**：
- `chunks` 仅存 L3 叶子块，包含向量，用于语义检索。
- `parent_chunks` 存 L1/L2 父块文本，用于 Auto‑merging 时快速获取父级内容。
- 其他集合均不包含向量，仅作结构化存储。Chroma 可以存储无向量的文档，只需在添加时传入空列表作为 embedding 即可。

### 3.2 元数据查询支持
Chroma 支持按 metadata 过滤，例如：
- 查询某个会话的所有消息：`collection.get(where={"session_id": "xxx"})`
- 查询某个文档的所有分块：`collection.get(where={"doc_id": "xxx"})`

### 3.3 检索流程（语义为主）
由于 Chroma 原生是语义检索，若需要类似 BM25 的关键词能力，可考虑：
- 在 `chunks` 集合的 metadata 中加入关键词列表字段，检索时先通过语义检索，再对结果做二次过滤；或
- 接入轻量级 BM25 库（如 `rank_bm25`）作为补充，但无需存储到 Chroma。

本项目初期可仅使用语义检索 + Rerank 保证效果，后续根据需求增加关键词检索。

## 四、物理结构设计

### 4.1 技术选型

| 组件       | 选型                                     | 说明                                          |
| ---------- | ---------------------------------------- | --------------------------------------------- |
| 后端框架   | Flask`开发成本低`                        | 轻量，支持生成器流式响应                      |
| Agent 框架 | LangChain                                | 封装工具、记忆、流式生成                      |
| 数据库     | Chroma                                   | 统一存储向量与结构化数据，持久化到本地        |
| 嵌入模型   | 阿里云向量模型（如 `text-embedding-v2`） | 通过 dashscope SDK 调用，生成稠密向量         |
| 生成模型   | Qwen3.5（API 调用）                      | 通过阿里云灵积或本地部署，用于 Agent 生成答案 |
| 精排模型   | 可选 Jina Rerank（若需提升效果）         | 通过 API 调用                                 |
| 前端       | Vue 3 CDN + SSE                          | 无需构建，快速实现                            |
| 部署       | Docker + Gunicorn                        | 容器化，多 worker 处理并发                    |

> [!CAUTION]
>
> 为何选用flask而不是fastapi？
>
> ### 1.开发成本低，学习曲线平缓
>
> - 团队成员对 Flask 更为熟悉，可以快速上手，无需额外学习异步编程（`async/await`）以及 FastAPI 特有的依赖注入系统。
> - Flask 是同步框架，代码编写风格更接近传统 Python 脚本，适合学生团队在短期内完成功能迭代。
>
> ### 2. 满足项目需求，性能足够
>
> - 本项目的并发量预期较低（校内演示、用户测试阶段），Flask 配合 Gunicorn 多进程足以支撑数百级并发。
> - 流式输出（SSE）在 Flask 中可通过生成器 + `Response` 轻松实现，无需依赖异步特性。
>
> ### 3. 生态成熟，资料丰富
>
> - Flask 拥有大量成熟的扩展（如 Flask-CORS、Flask-SQLAlchemy 等），遇到问题易于查找解决方案。
> - 学校实验室或学生项目中 Flask 的使用率更高，便于经验传承。
>
> ### 4. 避免过度设计
>
> - FastAPI 的异步优势在高 I/O 密集场景下明显，但本项目核心瓶颈在于 LLM API 调用和向量检索，这些操作本身就是 I/O 阻塞的，即便使用异步也需配合线程池，优势有限。
> - 采用 Flask 可以避免引入复杂的异步编程心智负担，将精力聚焦于 RAG 核心逻辑。

> [!WARNING]
>
> 为什么选择 Chroma 而非传统关系型数据库 + Milvus？
>
> ### 1. **简化架构，减少运维负担**
>
> - 传统方案需要维护三套存储：关系数据库（如 PostgreSQL）、向量数据库（Milvus）、以及用于缓存的 Redis。对于学生项目，这无疑增加了部署、配置、数据一致性维护的难度。
> - Chroma 作为一个“嵌入式向量数据库”，以文件形式持久化，无需单独部署服务，开箱即用，极大降低了运维成本。
>
> ### 2. **统一存储，消除数据冗余**
>
> - 在 Chroma 中，我们通过不同的 Collection 分别存储文档分块（含向量）、会话、消息、用户等数据。所有数据都在同一个数据库实例中管理，便于备份和迁移。
> - 传统方案中，文档分块需要同时存储向量到 Milvus、元数据到关系库，若想实现 Auto‑merging 还需额外存储父块，数据分散且同步复杂。
>
> ### 3. **满足项目规模与查询需求**
>
> - Chroma 支持对 metadata 的过滤查询（如 `where={"session_id": "xxx"}`），足以满足我们检索会话历史、用户信息等结构化需求。
> - 向量检索性能对于数千级文档分块完全足够，且 Chroma 底层使用 SQLite + HNSW 索引，检索速度快，资源占用低。
>
> ### 4. **LangChain 原生集成**
>
> - LangChain 提供了 `Chroma` 的官方 VectorStore 实现，可以直接与 LangChain 的 RetrievalQA 链、Memory 等组件无缝配合，减少适配工作量。
> - 而 Milvus 需要额外配置连接、集合管理，且 LangChain 对其支持相对复杂。
>
> ### 5. **成本与部署友好**
>
> - Chroma 为开源项目，不依赖外部云服务，可完全部署在校内服务器或本地笔记本上，便于演示和答辩。
> - 无需申请云资源或配置复杂的集群，符合学生项目对成本可控的要求。

| 对比维度   | 备选方案（FastAPI + 关系库 + Milvus） | 最终方案（Flask + Chroma）            |
| :--------- | :------------------------------------ | :------------------------------------ |
| 开发难度   | 高，需熟悉异步编程、多数据库交互      | 低，同步开发，单一数据库              |
| 运维成本   | 高，需维护三个独立组件                | 低，仅需部署 Flask 应用与 Chroma 文件 |
| 性能       | 理论上更高，但对本项目无明显优势      | 满足需求，检索与生成延迟可控          |
| 团队掌握度 | 部分成员不熟悉                        | 全部成员熟悉                          |
| 项目展示   | 配置复杂，易出错                      | 部署简单，便于现场演示                |

### 4.2 部署架构

```
[浏览器] → Nginx (静态资源 + 反向代理) → Gunicorn (Flask app)
                                         ↓
                               Chroma 数据库（持久化目录）
                               （无其他外部依赖）
```

### 4.3 索引与持久化
- Chroma 数据存储在本地目录（如 `./chroma_data`），启动时自动加载。
- 文档分块时，生成向量并写入 `chunks` 集合；父块写入 `parent_chunks` 集合。
- 会话、消息等结构化数据直接写入对应集合，无需向量化。

## 五、实施路径

### 5.1 阶段一：基础框架搭建（2周）
- **任务**：
  - 搭建 Flask 项目结构，配置路由。
  - 集成 Chroma，定义全局 `ChromaClient`，初始化各个 Collection。

- **产出**：可对话的简单聊天 Demo。

### 5.2 阶段二：文档处理与向量化（3周）
- **任务**：
  - 开发文档上传接口（`/documents/upload`）。
  - 实现三级分块（L1/L2/L3），将 L1/L2 父块存入 `parent_chunks` 集合（无向量）。
  - 集成阿里云向量模型 SDK（dashscope），为 L3 叶子块生成稠密向量。
  - 将 L3 叶子块（含向量、文本、元数据）存入 `chunks` 集合。
  - 实现文档删除功能：删除 `chunks` 和 `parent_chunks` 中对应的记录。
- **产出**：文档入库流程完整，支持检索。

### 5.3 阶段三：RAG 核心流程（4周）
- **任务**：
  - 实现检索函数：从 `chunks` 集合执行相似性检索（`collection.query`），返回 top_k 候选。
  - 实现 Auto‑merging：对检索结果中分数低于阈值的块，从 `parent_chunks` 集合读取父块替换。
  - 实现评分门控：LLM 判断检索结果相关性。
  - 实现查询重写（Step‑Back / HyDE）。
  - 可选：接入 Rerank API 提升排序质量。
  - 将以上流程封装为 LangChain 工具 `search_knowledge_base`。
- **产出**：RAG pipeline 可独立运行，输出高质量上下文。

### 5.4 阶段四：Agent 集成与流式输出（2周）
- **任务**：
  - 创建 LangChain Agent，绑定知识库工具和天气工具（可选）。

  - 实现跨线程步骤推送：使用 `queue.Queue` + 后台线程，通过 SSE 发送检索步骤。
- **产出**：完整智能问答，支持实时步骤展示。

## 七、与 Qwen3.5 和阿里云向量模型的集成细节

### 7.1 阿里云向量模型调用示例（使用 dashscope）
```python
import dashscope
from dashscope import TextEmbedding

def get_embedding(text):
    resp = TextEmbedding.call(
        model='text-embedding-v2',  # 或其他模型
        input=text,
        api_key='your-api-key'
    )
    return resp['output']['embeddings'][0]['embedding']
```

### 7.2 Qwen3.5 模型调用（使用阿里云灵积或 OpenAI 兼容接口）
```python
from langchain_community.chat_models import ChatOpenAI

llm = ChatOpenAI(
    model="qwen3.5",
    openai_api_key="your-api-key",
    openai_api_base="https://dashscope.aliyuncs.com/compatible-mode/v1",
    streaming=True
)
```

### 7.3 LangChain 集成 Chroma
```python
from langchain_community.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings  # 需自定义适配阿里云

# 需自行实现一个 Embeddings 类，调用阿里云 API
class AliyunEmbeddings(Embeddings):
    def embed_documents(self, texts):
        return [get_embedding(t) for t in texts]
    def embed_query(self, text):
        return get_embedding(text)

vectorstore = Chroma(
    collection_name="chunks",
    embedding_function=AliyunEmbeddings(),
    persist_directory="./chroma_data"
)
```

## 八、总结

本方案采用 **Chroma 作为唯一数据库**，统一管理结构化数据与向量，大幅降低系统复杂度；生成模型使用 **Qwen3.5**，嵌入模型使用 **阿里云向量模型**，满足低成本、易获取的要求；后端使用 **Flask**，轻量高效；前端使用 **Vue 3 CDN**，快速开发。整体架构清晰，适合学生大创项目展示与迭代。

按照上述实施路径分阶段推进，可确保项目按时完成，并产出具有实际应用价值的 RAG 智能问答系统，为大创项目“基于AIGC的OTA产品用户使用体验研究与产品优化设计”提供坚实的技术支撑。







# 用户实际使用时的完整底层流程

当真实用户通过前端页面提问时，系统将按照以下流程运作。假设后端采用 **Flask + Chroma + Qwen3.5**，并实现了 **RAG（检索增强生成）** 与 **流式输出（SSE）**。

---

## 一、用户交互阶段（前端）

1. 用户在聊天框中输入问题，点击“发送”。
2. 前端创建 `AbortController`（用于中断），向 `/chat/stream` 发送 POST 请求，请求体中包含：
   - `message`：用户问题文本
   - `session_id`：当前会话标识（若无则新建）
   - `user_id`：用户标识（用于区分不同用户）
3. 前端同时准备一个**空的消息气泡**，状态设为“思考中”，并开始等待 SSE 事件。

---

## 二、后端接收与初始化（Flask 路由）

1. Flask 接收到 `/chat/stream` 请求，解析 JSON 参数。
2. 创建一个 `queue.Queue`（线程安全队列），用于在后台线程和主生成器之间传递事件（流式输出单元）。
3. 调用一个工具函数，将队列注册到全局（供 RAG 工具函数使用，以便推送实时步骤）。
4. 启动一个**后台线程**，该线程负责运行 LangChain Agent 的流式生成，并将生成过程中的 token 和 RAG 步骤放入队列。
5. 主线程立即返回一个 `Response`，使用 `stream_with_context` 包装一个生成器，该生成器会**不断从队列中取事件**，并格式化为 SSE 事件（`data: {...}\n\n`）返回给前端。

---

## 三、RAG 核心流程（后台线程）

后台线程执行 LangChain Agent，当 Agent 决定调用 `search_knowledge_base` 工具时，会触发以下 RAG 流水线：

### 1. 检索阶段（Retrieval）

- **稠密向量检索**：  
  将用户问题通过阿里云向量模型生成稠密向量，调用 Chroma 的 `collection.query` 方法，在 `chunks` 集合中检索与问题最相似的 **L3 叶子块**，返回 top_k（例如 20 个）结果，每个结果包含 `chunk_id`、文本、相似度分数和元数据。

> [!TIP]
>
> 在评估RAG性能时：
>
> 1. **用户问题不是切块的**
>    在检索时，用户问题（原始 query）会直接生成一个查询向量（或用于 BM25 的查询词），然后去向量库中检索**文档分块**（chunks）。用户问题本身不会被切块，而是作为一个整体去匹配知识库里的块。
> 2. **“检索到的相关块数” 是指**
>    从向量库返回的候选块列表中，**与人工标注的正确答案块（ground truth）相匹配的那些块的数量**。
>    - 候选块是检索器返回的前 k 个结果。
>    - 人工标注的答案块是你事先为每个测试问题准备好的、应该被检索到的正确文档块（通常来自知识库中的某个或某几个 chunk）。
>    - 然后统计：候选块中命中了多少个这些标注块，这个数就是“检索到的相关块数”。
>
> **举个例子**：
>
> - 你为某个问题标注了 3 个正确的文档块（总相关块数 = 3）。
> - 检索器返回 top-5 候选块，其中包含 2 个你标注的块。
> - 那么 Recall@5 = 2/3 ≈ 0.67，Precision@5 = 2/5 = 0.4。
>
> **注意**：评估是在**离线测试集**上进行的，测试集中的每个问题都预先有人工标注好的“正确块列表”。在线运行时，用户看不到这些标注，评估只是用来衡量检索模块的质量。
>
> 所以，简单来说：**检索到的相关块数 = 系统返回的块 ∩ 人工标注的正确块**。

- **（可选）稀疏检索/关键词补充**：  
  若系统实现了 BM25 或类似关键词检索，则并行执行，再将两路结果通过 RRF 融合，取 top_k。
- **精排（Rerank）**：  
  将检索到的候选块与用户问题一起调用 Jina Rerank API，获取更精确的排序分数。
- **Auto‑merging**：  
  遍历候选块，若某块的 rerank 分数低于设定阈值（如 0.7），则从 `parent_chunks` 集合中查找其父块（L2 或 L1），用父块文本替换当前块。该过程可递归向上合并，但通常只合并一次。
- **上下文构建**：  
  将所有最终块（可能是原始叶子块或合并后的父块）的文本去重后拼接，作为后续生成的上下文。

### 2. 评分门控（Grade Documents）

- 将用户问题和上一步生成的上下文拼接成一个 Prompt，调用 LLM（Qwen3.5）判断上下文是否足以回答问题。
- 如果判断为“是”，则进入生成阶段；如果判断为“否”，则触发**查询重写**。

### 3. 查询重写（Query Rewriting）

- 根据问题类型（或随机策略）选择重写方式：
  - **Step‑Back**：让 LLM 生成一个更抽象的上位问题。
  - **HyDE**：让 LLM 生成一段假设性答案（虚构文档）。
- 将重写后的查询（或假设文档）作为新的查询，重复上述检索流程，获取**二次召回结果**。
- 二次召回的结果与初次召回结果合并去重，作为最终上下文。

### 4. 最终上下文准备

- 将合并后的上下文限制在一定的 token 长度内（如 2000 tokens），避免超出模型输入限制。
- 将上下文与对话历史（记忆）整合，形成完整的系统 Prompt。

---

## 四、Agent 生成答案（流式输出）

1. LangChain Agent 将上下文作为工具返回结果，开始生成最终答案。
2. 生成过程使用 `llm.stream`（或通过 `agent.astream`），每次产生一个 token（文本片段）。
3. 这些 token 被包装为 `{"type": "content", "data": token}` 格式，放入队列。
4. **实时步骤推送**：在检索、评分、重写的每个关键节点，工具函数会调用 `emit_rag_step`，将步骤信息（如 `{"type": "rag_step", "step": {...}}`）也放入同一个队列。
5. 后台线程持续将 token 和步骤放入队列，直到生成结束，最后放入 `None` 作为结束标记。

---

## 五、主线程的 SSE 生成器

主线程的生成器循环：

```python
def generate():
    while True:
        event = queue.get()  # 阻塞直到有事件
        if event is None:
            break
        yield f"data: {json.dumps(event)}\n\n"
```

- 当队列中取到 `rag_step` 事件时，立即发送到前端，前端实时更新“思考步骤”文字。
- 当取到 `content` 事件时，前端将 token 追加到当前消息气泡中，实现打字机效果。
- 前端一旦收到第一个 `content` 事件，就会将气泡从“思考中”切换为“回答中”。

---

## 六、异常处理与终止

- **用户主动终止**：前端调用 `AbortController.abort()`，浏览器中断请求。Flask 的生成器会收到 `GeneratorExit` 异常，此时主线程会取消后台线程（通过设置标志或直接取消），并停止生成。
- **后端异常**：若生成过程中出现异常（如 API 超时），后台线程会将错误事件放入队列，前端显示友好错误提示。

---

## 七、数据持久化

- **消息存储**：当完整回答生成后，后台线程将用户问题和 AI 回答分别存入 Chroma 的 `messages` 集合（不含向量），元数据包含 `session_id`、`role`、`timestamp` 等。
- **会话更新**：若会话标题为空，则用用户问题的前几个字作为标题。
- **检索日志**：将本次检索的查询、命中块列表、耗时等信息存入 `retrieval_logs` 集合，用于后续分析与优化。

# 用户输入问题的向量化与相似度检索过程详解

在基于 RAG 的问答系统中，用户输入的问题会经历**向量化**和**相似度检索**两个关键步骤，才能从知识库中找到最相关的文档片段。下面结合本项目使用的 **Flask + Chroma + 阿里云向量模型** 技术栈，详细说明底层原理与实现。

---

## 一、用户问题的向量化

### 1.1 什么是向量化？
向量化是将文本（如用户问题）转换为一组浮点数数组（即**向量**）的过程。这个向量能够捕获文本的语义信息，使得语义相近的文本在向量空间中的距离也更近。

### 1.2 项目中如何实现向量化？
本项目使用**阿里云向量模型**（如 `text-embedding-v2`）完成向量化。该模型通过 API 调用，将输入文本转化为 1024 维的稠密向量。

**核心步骤**：
1. 前端将用户问题发送到后端 Flask 接口。
2. 后端接收到问题文本后，调用阿里云嵌入模型 API。
3. API 返回一个向量（列表形式的浮点数）。

**示例代码（简化）**：
```python
import dashscope
from dashscope import TextEmbedding

def embed_text(text):
    resp = TextEmbedding.call(
        model='text-embedding-v2',
        input=text,
        api_key='your-api-key'
    )
    # 提取向量
    embedding = resp['output']['embeddings'][0]['embedding']
    return embedding  # 返回类似 [0.123, -0.456, ...] 的列表
```

> **注意**：实际项目中我们会将上述函数封装为 LangChain 兼容的嵌入类，但原理相同。

---

## 二、相似度检索：如何找到最相关的文档块？

### 2.1 检索流程概览
1. **知识库准备**：在系统启动或文档上传时，已将文档切分为小块（chunks），并对每个块调用相同的嵌入模型生成向量，存储到 Chroma 数据库中。
2. **用户提问时**：将用户问题的向量作为**查询向量**，与 Chroma 中存储的所有文档块向量进行相似度计算。
3. **返回最相似的 k 个块**：Chroma 默认使用**余弦相似度**（cosine similarity）或**内积**（dot product）计算两个向量的相似度，并按照相似度从高到低排序，返回 top_k 个文档块及其相似度分数。

### 2.2 Chroma 内部的相似度计算机制
Chroma 底层使用 **SQLite** 存储元数据，并使用 **HNSW** 索引加速向量检索。当执行 `collection.query()` 时，内部会：

- 将查询向量与索引中的向量进行近似最近邻搜索。
- 默认相似度度量是 **余弦相似度**（通过归一化后计算内积实现）。
- 返回每个候选块的相似度分数（范围通常在 0~1 之间，越大表示越相似）。

### 2.3 代码实现（在项目中使用）
```python
from langchain.vectorstores import Chroma

# 假设已初始化 vectorstore，并配置了嵌入模型
vectorstore = Chroma(
    collection_name="chunks",
    embedding_function=aliyun_embeddings,  # 自定义嵌入类
    persist_directory="./chroma_data"
)

# 用户提问时
query = "北海有哪些必去景点？"
# 1. 向量化（由 embedding_function 自动完成）
# 2. 检索最相似的 5 个文档块
results = vectorstore.similarity_search_with_score(query, k=5)

for doc, score in results:
    print(f"相似度: {score:.4f}\n内容: {doc.page_content}\n")
```

**解释**：
- `similarity_search_with_score` 方法会：
  - 调用 `embedding_function.embed_query(query)` 得到查询向量。
  - 将该向量与 Chroma 索引中的向量进行比对，返回最相似的 k 个文档对象及其相似度分数。
- 返回的 `doc` 是 LangChain 的 `Document` 对象，包含 `page_content`（文本）和 `metadata`（元数据）。

### 2.4 相似度分数的含义
- **余弦相似度**：值接近 1 表示向量方向几乎一致（语义高度相关），接近 0 表示无关。
- 在 Chroma 中，分数通常为相似度值（直接来自索引计算），而不是距离。你可以根据分数阈值过滤低质量结果。

## 三、完整的用户问题处理链路（结合项目）

1. **用户在前端输入问题** → 发送到 `/chat/stream`。
2. **Flask 路由接收** → 启动后台线程运行 LangChain Agent。
3. **Agent 决定调用知识库工具** → 工具内部调用 RAG 流程。
4. **RAG 流程中的检索步骤**：
   - 调用 `vectorstore.similarity_search_with_score(query, k=20)` 获得初步候选块及分数。
   - （可选）对候选块进行 Rerank 精排。
   - 根据分数和 Auto‑merging 规则，决定是否用父块替换低分块。
   - 合并去重，形成最终上下文。
5. **上下文传递给生成模型** → 流式返回答案。

## 四、常见问题与注意事项

### Q1：为什么用户问题也要向量化？直接用原始文本检索不行吗？
- 原始文本无法直接与向量库中的向量比较。只有将查询也映射到同一个向量空间，才能通过数学计算（如点积）衡量语义相似度。

### Q2：Chroma 返回的相似度分数是余弦相似度吗？
- 是的，默认情况下 Chroma 使用余弦相似度（通过内积计算，向量已归一化）。你可以在创建集合时指定不同的度量（如 L2 距离）。

### Q3：如果用户问题很长，向量化会有什么影响？
- 嵌入模型通常有输入长度限制（如 512 token）。长文本可能会被截断或导致语义稀释。建议在检索前对用户问题进行简单压缩或保留关键部分，但大多数情况下直接向量化即可。

### Q4：如何提高检索质量？
- 使用**混合检索**（向量检索 + 关键词检索）并融合排序。
- 使用**Rerank 模型**（如 Jina Rerank）对初筛结果重新打分。
- 调整分块策略，保证文档块语义完整且大小适中。
- 根据业务场景调优相似度阈值。

通过以上步骤，用户问题被转化为向量，并与知识库中预先生成的文档向量进行相似度匹配，从而快速定位最相关的信息片段，为后续生成高质量答案奠定基础。
