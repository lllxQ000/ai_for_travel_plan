# SuperMew RAG 项目的运维空间

作为学生大创项目，运维并非核心目标，但为了保障系统稳定运行、便于后续优化和展示，我们规划了以下运维空间，涵盖数据、配置、监控、性能、部署等方面。

---

## 一、数据存储运维

### 1. Chroma 向量数据库
- **备份与恢复**  
  Chroma 数据以文件形式存储在本地目录（如 `./chroma_data`）。运维脚本定期（如每日）将此目录压缩备份到安全位置，防止磁盘损坏或误删导致数据丢失。  
- **清理过期数据**  
  当文档被删除时，需同步删除 Chroma 中对应的向量集合（`doc_chunks`、`parent_chunks`）。我们已在代码中实现删除接口，运维人员可定期检查孤立数据并手动清理。
- **性能监控**  
  监控 Chroma 目录大小、检索响应时间，当数据量过大时考虑分片或迁移到更强硬件。

### 2. 会话存储（JSON 文件）
- **定期归档**  
  用户会话 JSON 文件会随时间增长。运维脚本每月自动压缩历史会话，保留最近三个月数据，其余归档至冷存储。
- **并发写保护**  
  目前 JSON 文件在高并发下可能存在写入冲突。后续可引入文件锁或迁移至 SQLite，运维时需关注文件锁情况，必要时扩展。

---

## 二、配置与环境变量管理

- **环境变量集中管理**  
  所有敏感信息（API 密钥、Milvus 地址、Rerank 端点等）均通过 `.env` 文件加载。运维人员需确保不同环境（开发、测试、演示）使用独立配置文件，避免生产密钥泄露。
- **配置热更新**  
  部分参数（如 `AUTO_MERGE_THRESHOLD`、`LEAF_RETRIEVE_LEVEL`）通过环境变量控制，修改后需重启应用。运维时可通过 `supervisor` 或 `systemd` 实现平滑重启。

---

## 三、日志与监控

### 1. 应用日志
- **结构化日志**  
  使用 `logging` 模块记录请求 IP、用户 ID、会话 ID、检索耗时、模型调用耗时、异常堆栈等。日志级别分为 INFO（常规请求）、WARNING（降级）、ERROR（故障）。
- **日志轮转**  
  配置日志文件大小限制和轮转策略（如每天或 100MB），防止磁盘占满。

### 2. 关键指标监控
- **API 响应时间**  
  通过中间件记录 `/chat/stream` 的首字节时间和总时间，存入监控系统（如 Prometheus 或简单记录到日志）。
- **检索质量**  
  记录每次检索的召回块数量、rerank 是否成功、自动合并触发次数等，定期分析日志，调优参数。
- **模型 API 调用统计**  
  统计每日嵌入和 LLM 调用次数，估算成本，设置报警阈值。

### 3. 健康检查
- 提供 `/health` 端点，返回 200 表示应用正常；同时检查 Chroma 可访问性、API 密钥有效性，确保依赖服务可用。

---

## 四、性能与并发运维

- **Gunicorn worker 调优**  
  Flask 应用部署时使用 Gunicorn，根据服务器 CPU 核心数设置 worker 数量（如 2*CPU+1）。观察 CPU 和内存使用，动态调整。
- **线程池大小**  
  后台线程用于运行 Agent 流式生成，需根据并发请求数设置合适的线程池上限，避免资源耗尽。使用 `concurrent.futures.ThreadPoolExecutor` 管理。
- **数据库连接池**  
  Chroma 本身无连接池概念，但若后期引入 SQLite，需配置连接池大小。

---

## 五、故障恢复与降级

- **检索降级**  
  当混合检索失败时，代码已实现降级为纯稠密检索；若全部失败，返回空结果并记录错误。
- **Rerank 降级**  
  Rerank API 不可用时，自动跳过精排，直接使用 RRF 融合结果。
- **LLM 调用失败**  
  若生成模型 API 超时或报错，返回友好提示，不中断服务。
- **自动重启**  
  使用 `systemd` 或 `supervisor` 监控 Flask 进程，异常退出时自动重启。

---

## 六、部署与版本管理

- **容器化部署**  
  提供 Dockerfile，将 Flask 应用、Chroma 数据目录、依赖环境打包为镜像。使用 Docker Compose 一键启动，便于在不同服务器上复现环境。
- **版本控制**  
  使用 Git 管理代码，每次功能迭代打 tag。运维时通过 `git pull` 更新代码，并执行数据库迁移脚本（如有）。
- **回滚机制**  
  保留最近三个版本的 Docker 镜像和配置文件，出现问题时可快速回滚。

---

## 七、安全与合规

- **API 密钥保护**  
  所有密钥仅通过环境变量注入，禁止硬编码。生产环境使用密钥管理服务（如 Docker Secrets）。
- **用户数据隔离**  
  目前未实现多用户权限隔离，若后期扩展，需在数据库层面添加用户 ID 过滤，防止跨用户数据泄露。
- **HTTPS 支持**  
  部署时配置 Nginx 反向代理，启用 HTTPS，保护会话传输安全。

---

## 八、成本控制

- **模型 API 调用限流**  
  在代码层面对嵌入和 LLM 调用添加限流器（如每秒最多 10 次），避免因恶意请求导致费用超支。
- **缓存策略**  
  对相同或相似问题设置缓存（如 Redis），短期内直接返回历史答案，减少重复 API 调用。

---

- **团队知识传递**  
  定期分享运维经验，确保后续成员能接手维护。
