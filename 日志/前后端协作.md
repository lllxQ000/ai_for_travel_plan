# 前端表单 → 后端接收：如何设计“同程式”定制规划助手

你想做一个类似“同程 AI 定制路线规划师”的界面：用户点选几个选项（天数、人数、风格等），后端就能接收这些参数，然后通过 RAG 生成个性化行程。这种交互的核心是**前端将表单数据序列化成 JSON，后端按约定字段名解析**。下面分步讲解如何设计，并给出前后端协作的最佳实践。

---

## 一、前端表单设计

### 1. 使用 Vue 3 的响应式数据绑定
在 `frontend/index.html` 中，用 `v-model` 绑定表单控件的值。

```html
<div id="app">
  <div class="plan-form">
    <label>目的地：</label>
    <input v-model="planData.destination" placeholder="例如：桂林" />

    <label>出行天数：</label>
    <select v-model="planData.days">
      <option>1天</option>
      <option>2天1晚</option>
      <option>3天2晚</option>
      <option>4天3晚</option>
    </select>

    <label>出行类型：</label>
    <select v-model="planData.travelType">
      <option>个人游</option>
      <option>情侣/夫妻</option>
      <option>家庭游（带小孩）</option>
      <option>朋友结伴</option>
    </select>

    <label>游玩风格：</label>
    <div>
      <label><input type="radio" value="轻松" v-model="planData.style"> 轻松（不赶路）</label>
      <label><input type="radio" value="紧凑" v-model="planData.style"> 紧凑（多打卡）</label>
      <label><input type="radio" value="文化深度" v-model="planData.style"> 文化深度</label>
    </div>

    <label>特殊偏好（多选）：</label>
    <div>
      <label><input type="checkbox" value="摄影" v-model="planData.prefs"> 摄影</label>
      <label><input type="checkbox" value="美食" v-model="planData.prefs"> 美食</label>
      <label><input type="checkbox" value="徒步" v-model="planData.prefs"> 徒步</label>
    </div>

    <button @click="submitPlan">生成行程</button>
  </div>
</div>
```

### 2. Vue 组件中的数据与提交方法
```javascript
const { createApp, ref } = Vue;

createApp({
  setup() {
    const planData = ref({
      destination: '',
      days: '3天2晚',
      travelType: '个人游',
      style: '轻松',
      prefs: []
    });

    const submitPlan = async () => {
      const response = await fetch('/api/plan', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(planData.value)
      });
      // 处理流式响应...
    };
    return { planData, submitPlan };
  }
}).mount('#app');
```

这样前端发送的 JSON 格式就是：
```json
{
  "destination": "桂林",
  "days": "3天2晚",
  "travelType": "个人游",
  "style": "轻松",
  "prefs": ["摄影", "美食"]
}
```

## 二、后端接收：定义 Pydantic 模型

在 `backend/schemas.py` 中，创建一个模型与前端字段名**完全一致**（也可以使用别名）。

```python
from pydantic import BaseModel
from typing import List

class PlanRequest(BaseModel):
    destination: str
    days: str
    travelType: str
    style: str
    prefs: List[str] = []   # 可选偏好
```

然后在 `app.py` 中添加路由：

```python
from backend.schemas import PlanRequest

@app.post('/api/plan')
def plan_stream(req: PlanRequest):
    # 将结构化的用户偏好转成自然语言查询
    query = build_query_from_plan(req)
    # 调用流式生成（复用之前的 RAG 流程）
    return Response(generate_stream(query), mimetype='text/event-stream')
```

其中 `build_query_from_plan` 可以把字段拼接成一段自然语言，例如：
```python
def build_query_from_plan(plan: PlanRequest) -> str:
    return f"帮我规划一个{plan.destination}{plan.days}的行程，{plan.travelType}，偏好{plan.style}风格，喜欢{', '.join(plan.prefs)}。"
```

## 三、开发顺序：先约定接口契约

**不需要先写前端再写后端**，而是**同时并行**，前提是双方约定好接口格式（字段名、类型、是否必填）。你可以：

1. **先定义接口文档**（或直接定义 Pydantic 模型作为“契约”）。
2. 前端根据契约写表单，保证提交的字段名、类型匹配。
3. 后端根据契约实现路由，并立即可以用模拟数据测试。
4. 之后前后端联调。

这种方式称为“契约驱动开发”，可以避免后期字段名不一致的麻烦。

---

## 四、完整示例：结合 RAG 流式输出

沿用你已有的 `/chat/stream` 接口，可以将表单数据转换为自然语言后，复用同一套 Agent 流程。

```python
@app.post('/api/plan')
def plan_stream(req: PlanRequest):
    query = build_query_from_plan(req)
    # 直接调用现有的流式聊天接口，只是把消息换成 query
    return chat_stream(ChatRequest(message=query))
```

这样，前端提交表单后，后端自动构造问题，进入已有的 RAG + 流式生成逻辑，就能输出定制行程。

---

## 五、总结

- **前后端协作**：前端用 `v-model` 收集数据，`fetch` 发送 JSON；后端用 Pydantic 模型自动解析并校验。
- **字段名约定**：前后端使用相同的字段名（如 `destination`、`days`），或后端通过 `alias` 映射。
- **开发顺序**：先定义接口契约（Pydantic 模型），然后前后端并行实现。
- **复用现有能力**：只需将表单数据转成自然语言查询，就能无缝接入你已有的 RAG 系统。

这种设计既保持了前端交互的友好性，又充分利用了后端已经实现的智能检索与生成能力。下一步你可以着手制作前端表单，并定义好 PlanRequest 模型，即可快速完成“同程式”定制规划助手。
